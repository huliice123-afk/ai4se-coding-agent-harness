# SPEC_PROCESS.md — 规约与计划生成过程记录

## 1. brainstorming 关键节点

### 1.1 初始方向确认

在与 Superpowers 的 brainstorming 过程中，智能体首先追问了项目类型选择。A 方向（Coding Agent Harness）是课程首选，我选择了该方向，因为 harness 的工程深度（决策/工具/记忆/治理/反馈/配置六个维度）最契合课程目标"LLM 能完成编码工作时，工程师的价值在哪里"。

### 1.2 技术栈锁定

智能体提出 Java + Claude API 的组合。我决定使用 Java（熟悉语言）+ Claude（编码能力最强），开发阶段使用 DeepSeek/GLM 降低成本。智能体追问了构建工具（Maven）、测试框架（JUnit 5 + Mockito）、HTTP 客户端（OkHttp）等细节，全部写入 AGENTS.md 作为项目约束。

### 1.3 重点维度选择

智能体列出了六个维度并建议选一个深入。文档指出治理、反馈闭环、扩展三个方向"天然由代码构成，深入实现后最能体现工程深度"。结合 300 元预算限制，反馈闭环因开发成本低、可单测性强被选中。

### 1.4 工具集与记忆设计

智能体逐步确认了四类工具（文件/Shell/Git/搜索）、三层护栏（命令/文件/网络）、Codex 风格的记忆系统（Markdown 文件 + 按需检索）。每个设计决策都经过简短确认，没有反复修订。

### 1.5 SPEC 自审

智能体在 SPEC 完成后进行了自审，发现了两个问题：
- CI 配置应使用 GitHub Actions 而非 GitLab CI（项目仓库在 GitHub）
- 缺少机制演示脚本的明确定义（§A.6 要求）

两项修复后写入 SPEC 并提交。

## 2. 关键迭代与决策

### 2.1 第一轮：AGENTS.md 约束

**AI 提出**：在 brainstorming 开始前先写 AGENTS.md 作为项目全局约束。
**我的决策**：采纳。决定包含技术栈、代码规范、项目纪律、安全约束。
**反思**：这是一个好建议。AGENTS.md 为后续所有 subagent 提供了统一的行为准则，避免了每个 task 都需要重复说明约束。

### 2.2 第二轮：模型使用策略

**AI 提出**：在 AGENTS.md 中增加模型切换策略——日常开发用 DeepSeek/GLM，复杂任务切换 Claude。
**我的决策**：采纳并补充到 AGENTS.md。
**反思**：300 元预算下，这个策略是必要的。实际开发中，大部分 task 是机械性编码（创建 POJO、写测试），DeepSeek 完全够用。

### 2.3 第三轮：分发方式

**AI 提出**：Docker 和 JAR 双分发。
**我的决策**：采纳。
**反思**：Docker 适合服务端部署，JAR 适合本地开发。双分发增加了工程复杂度但覆盖了更多场景。

### 2.4 第四轮：代码审查修复

**AI 审查发现**：HITL 流程未实现、MemoryRetriever 硬编码 key 名、AgentLoop 测试不足。
**我的决策**：全部采纳修复。
**反思**：代码审查是 Superpowers 流程中最有价值的一环。审查者发现了 3 个我自己无法注意到的问题，尤其是 HITL 的缺失（GuardResult 有 HITL 状态但从未被触发）。

## 3. AI 建议采纳与推翻

| AI 建议 | 采纳/推翻 | 原因 |
|---------|----------|------|
| 先写 AGENTS.md | 采纳 | 建立项目统一约束 |
| 模型切换策略 | 采纳 | 预算限制下的合理选择 |
| 反馈闭环作为重点 | 采纳 | 符合"机制必须是代码"的评分标准 |
| 双分发（Docker + JAR） | 采纳 | 覆盖更多使用场景 |
| 所有代码审查修复 | 采纳 | 实质性改进 |

无推翻的 AI 建议。在 brainstorming 过程中，AI 的每个建议都有合理的技术理由，且都经过我的确认后才执行。

## 4. 反思：brainstorming 技能的优缺点

**做得好的地方：**
- 一次一个问题，不会信息过载
- 每个设计决策都经过确认，不存在"AI 猜测我的意图"
- SPEC 自审机制有效（发现了 CI 配置和机制演示的缺失）

**让我不满的地方：**
- 过程偏慢。23 个 task 的 PLAN 需要大量 token 消耗
- 没有主动提醒冷启动验证（§4.5）。这个应该在 brainstorming 结束时提醒，但被跳过了

## 5. 冷启动验证

由于开发过程中网络限制（GitHub 连接需要代理），冷启动验证（用不同 agent 仅凭 SPEC+PLAN 试实现 1-2 个 task）未能在开发前完成。这是一个已知的流程偏离，在 AGENT_LOG.md 中记录了原因。

替代方案：代码审查（requesting-code-review）作为冷启动的替代，一个独立的审查 agent 发现了 4 个问题，起到了类似"不同视角发现 spec 缺陷"的作用。
