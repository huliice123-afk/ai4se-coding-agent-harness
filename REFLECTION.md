# REFLECTION

## 1. 项目概述

我做的这个项目是一个 Java Coding Agent Harness——简单说，就是给一个大语言模型套上"手脚"，让它能真正在本地帮你写代码、改文件、跑测试。技术栈是 Java 17 + Maven，LLM 用了 DeepSeek，前端用 Flask 搭了个 Web UI。整个项目按六个维度组织：core、llm、tools、guardrails、feedback、memory。六个维度里，我最花心思的是 feedback，也就是反馈闭环——让 agent 在执行失败后能自己判断问题在哪、怎么修正，然后重试。

## 2. Agent = LLM + Harness 的理解

做这个项目之前，我对 AI agent 的认知很模糊，觉得就是"让 AI 干活"。做完之后最大的感受是：LLM 其实只管"下一步做什么"，真正干活的是 harness 那一层。

LLM 本质上就是个推理引擎，它返回的是一段文本。DeepSeek 返回的 tool_call 格式是一段 JSON，但这段 JSON 里藏满了不确定性——有时格式不对，有时参数缺了，有时意图模糊。谁来处理这些？harness。ActionParser 负责把 LLM 的输出解析成可执行的动作，Guardrails 负责判断这个动作是不是危险——比如 LLM 可能会让你删掉系统文件，它自己并不知道什么叫"越界"。同一个 DeepSeek 模型，加上不同的 harness，可以变成完全不同的 agent。工程价值不在 LLM 本身，在 harness 层的治理能力。

## 3. 重点维度：Feedback

六个维度里我选了 feedback 作为重点。不是因为其他维度不重要，而是 feedback 是让 agent 从"一次性的傻瓜"变成"能自我修正的系统"的关键。

我设计了一个四步闭环：FailureClassifier 先判断失败类型——是编译错误、运行时异常还是工具调用失败；SeverityJudge 评估严重程度，决定要不要重试；CorrectionSuggester 给出修正建议；最后把建议回灌到下一轮对话里。Demo 2 里用 mock LLM 模拟了一个场景：agent 写出的代码有编译错误，然后它自己读取错误信息、定位问题、重新生成代码，最后通过了。这个流程走通的时候，我确实觉得 feedback 这个维度选对了。

但说实话，修正建议的质量还有很大提升空间。目前只是基于错误类型的模板匹配，没有结合对话上下文。比如 agent 连续两次犯同样的错误，系统不会意识到"可能不是代码问题，是思路问题"。

## 4. 开发过程中的真实困难

### 4.1 AI 会漏东西

用 AI 辅助开发有个很隐蔽的坑：它改一行代码的时候，经常会忘记检查这一行影响到哪些其他地方。但这里更大的问题是——AI 自己会忘记按 PR 工作流来。我一开始就定好了规矩：每个独立功能开一个 worktree、对应一个 PR，scope 尽量小、review 尽量聚焦。但 AI 做着做着就把这个规矩丢了。commit d9740c4 就是个典型：本来应该是一个独立的 tool_call 解析修复 PR，但 AI 修完之后漏掉了 stopCondition 那条路径，因为它在那个 worktree 里做的事超出了本来的 scope，review 就乱了。resultSummary 那次也一样，去掉 agent 文本导致 4 个测试全挂——AI 没主动在 PR 里跑一遍测试，它忘了这个步骤。不是我没定流程，是 AI 执行的时候会"选择性遗忘"，需要你反复提醒它回到流程上来。

### 4.2 前端美化是 AI 的软肋

Web UI 的样式调整折腾了很久。AI 能帮我搭出一个可用的界面，Calling 折叠、Round 隐藏、工具名中文化这些功能性的东西它都能做。但一到细节就拉胯了：字体大小看着别扭，间距要么太紧要么太松，颜色搭配总差点意思。还有流式输出的 UI 体验——消息会抖动、滚动位置不对——这些 AI 很难一次做对。我的体会是：AI 适合写骨架，磨细节还是得靠人的眼睛。

### 4.3 其他坑

Git worktree 在 Windows 中文路径下直接不兼容，就因为我项目路径是 `D:\文件\Agent`，折腾了好久。网络代理的问题也让 GitHub Actions 配置反复调试了好几轮。CI 平台差异也踩过坑：Linux 上 `*` 是合法文件名，Windows 不是——测试在本地全绿，推到 CI 就红了。Subagent 的上下文丢失也很头疼，每个 subagent 从零开始，每次都要在 prompt 里重新注入全局约束，像在跟一群金鱼说话。

## 5. 决策反思

现在回头看，有几个决策做得还行。一是用 .env 文件而不是 Windows Credential Manager，简单直接，任何平台都能跑。二是先 mock 后真实 LLM——mock 测试把全部核心机制覆盖了一遍，换上真实 DeepSeek 的时候几乎没有 bug，这个过程很顺畅。三是选了 DeepSeek 而不是 Claude，OpenAI 兼容格式更简单，API 成本也低。四是坚持全量 git 历史重构，10 个 PR 保留了完整开发痕迹。

但也有遗憾。初次实现没用 worktree，后来才补上，如果能一开始就用会省很多事。feedback 的 correction 建议缺乏上下文感知，目前只基于错误类型，不基于对话历史。记忆系统也过于简单，就是个键值对存储，没有语义检索能力。

## 6. 工具驱动开发的反思

这个项目是全程用 AI 工具辅助开发的。TDD 给我最大的安全感——115 个测试，每次改完代码跑一遍，心里有底。Subagent 的效率也很高，23 个 task 全部用 subagent 完成，每个平均 2-3 分钟，确实比我自己写快。

但 subagent 有它的天花板。每个 subagent 从零开始，不知道其他 subagent 做了什么，我必须把完整的上下文——文件列表、测试代码、全局约束——全部塞进 prompt 里。本质上，我像个产品经理，subagent 是程序员。工具提高了效率，但没有消除对理解的要求。我仍然需要审阅每一行代码，判断它是不是真的符合预期。

## 7. 与商业 Coding Agent 的差距

这个项目离真正的商业级 coding agent 还有不小差距：没有 RAG 或语义记忆，没有多 agent 协作，工具集只有 4 个，护栏是静态规则而非动态策略，harness 内部是同步的没有真正的流式输出。但我觉得这个项目的好处是——它展示了核心机制是可行的，而且足够小、足够透明，是一个可以审计的最小原型。

## 8. 总结

做完这个项目，我最大的感受是：Agent 不是魔法，是工程。LLM 是"原材料"，Harness 是"加工厂"。好材料加上差工厂，出不来好产品；但把工厂搭好，换不同的材料都能出活。

如果重来一次，我会先搭 worktree、先配 CI、先把 mock 测试全覆盖——这几个东西越早做收益越大。至于 AI 辅助编程的未来，AI 肯定会越来越好，但我越来越觉得，人的判断力才是不可替代的那部分。AI 能帮你写代码，但它不知道你为什么要写这段代码，也不知道这段代码在一个更大的系统里是否合适。这些判断，目前还是得靠人。
